JAMstack is a term coined by Mathias Biilman, the co-founder of Netlify. He came up with the term to help explain the types of websites his hosting company specializes in. Basically to assist in their marketing to better connect the concept with their audience.

So unlike other common web stacks like LAMP, MERN, LEMP (and of course there are many more) the JAMstack is not a list of specific technologies, but more of an architectural approach for building websites.

The official Jamstack.org website defines the JAMstack as‚Ä¶

An architecture designed to make the web faster, more secure, and easier to scale. It builds on many of the tools and workflows which developers love, and which bring maximum productivity.

The core principles of pre-rendering, and decoupling, enable sites and applications to be delivered with greater confidence and resilience than ever before.

The JAM in JAMstack

The JAM in JAMstack stands for:

  • JavaScript
  • APIs
  • Markup

JavaScript: Dynamic functionalities are handled by JavaScript. There is no restriction on which framework or library you must use.

APIs: All server-side processes or database actions are abstracted into reusable APIs, accessed over HTTPS with JavaScript. These can be custom-built or leverage third-party services.

Markup: Templated markup should be prebuilt during deployment, usually using a site generator or a build tool.

JAMstack Terms

Here are a few terms defines that are used in the JAMstack:

  • Decoupled
  • Pre-rendering
  • Progressively enhanced

Decoupled refers to the frontend being separated from the back end. The front end is typically built using a static site generator. The back end is often integrated with the front through the use of APIs used during the build process. Server-side processes can also be run using serverless functions.

While various practices exist for introducing dynamic elements to Jamstack sites, most are pre-rendered, which means the frontend was built and compiled into HTML, CSS, and JavaScript files.

JavaScript can be introduced to pre-rendered sites on an as-needed basis, thus increasing performance in the browser. Also some JAMstacks could be referred to as hybrid, where they have a mixture of static and server-side generated content or static content with client-side Javascript loading in dynamic data.

Why the Jamstack? ü§∑

Why use the Jamstack over other common stacks like the LAMP (Linux, Apache, MySQL & PHP) stack?

Here are a few of the the Benefits:

  • Security
  • Scaleable
  • Performance
  • Reliability
  • Portability & Flexibility
  • Developer Experience
  • Cost

Security

Without the typical server infrastructure, you no longer will need to worry about the security for the server‚Äôs software and applications. Most APIs are provided by hosted services, so they will be responsible for their security. Also since your front facing website will be composed of all static files, that reduces your security burden as well.

Scaleable

If your site suddenly goes viral and has a ton of active users, the CDN seamlessly compensates.

Performance

Pages will be served with pre-built markup and assets over a CDN. Most JAMstack specific hosting comes well tuned out-of-the box, so no worries about that.

Reliability

Since the sites are generated in a coded build process, if there are errors, the site will just not complete its build and will just show the current version of the site. Not to say frontend errors couldn‚Äôt be possible, but using this best practice of CI allows you to add linting and also can run automated tests for performance, SEO, Accessibility, Optimize images, Lighthouse audit, visual regression and more.

Portability & Flexibility

Since the sites are pre-rendered it means you can host them in a wide array of hosting services. Along with the portability, you can very easily move them to different hosting platforms and find the one you like best.

It essentially means you don‚Äôt have to worry about infrastructure lock-in.

Since the JAMstack is decoupled you have the ability to pick and choose sites and frameworks you are most comfortable with and also the best in-class services to power your site.

Developer Experience

Since you can use a large variety of tools to build your Jamstack site, it can make it easier to find one that fits your skill set.

Front end developers can focus more on the front end, without being tied to a monolithic architecture. This typically translates into faster and more focused development.

Also the JAMstack‚Äôs modern structure utilizes a common set of best practices that most developers share. This includes:A git workflow, Automating some of the development process with continuous integration and continuous deployments (CI/CD), The usage of Modern Build Tools like Babel, PostCSS, Webpack.

Cost

Most sites/services used in the JAMstack either offer a free or low cost plan.

Depending on the specific technologies you choose for your JAMstack site, you can easily build it completely for free (minus the cost for a domain name - I haven‚Äôt found a completely free option there).

Coding is typically required: If you want to make any layout or design updates on your websites or landing pages, you will need the help of a qualified developer.

The Initial Setup Time Overall time to build a new website is greater. Until you build at least one or two JAMstack sites and get a good base you can reuse (the technology and experience), it‚Äôll take some time to spin one up compared to a typical WP site.

Small plugin ecosystem for Site GeneratorsHowever with Javascript you can use almost any NPM package to add functionality to your website, so if you count that it has a MASSIVE collection, however it still isn‚Äôt the same as just adding a WP plugin.

Build times In most cases, JAMstack sites require a build time that could take a minimum of 2-5 minutes or even much longer depending on the amount of pages and images included in the website. That being said, there are services and Frameworks that can speed up the process a lot. Gatsbyjs.com‚Äôs hosting provides incremental builds in the cloud that can reduce build times from minutes to down to seconds. Also with NextJS you can use Incremental Static Regeneration (ISR). It‚Äôs a hybrid solution of SSG and SSR. It enables developers and content editors to use static-generation on a per-page basis, without needing to rebuild the entire site. With ISR, you can retain the benefits of static while scaling to millions of pages. The page is statically generated on the first request. Unlike in SSR, where the visitor has to wait for the data fetching, a fallback page is served immediately. With adding a Regeneration time (in seconds) NextJS will re-fetch the data to see if there‚Äôs any changes and if so it‚Äôll update the page.

PreviewsSome JAMstack setups do not provide instant previews. However some do include instant or almost instant previews of content changes.

Cost (yea it can be a con too if you aren‚Äôt careful) Microservices typically offer free or low cost monthly plans, however be aware some services can get pricey after that free plan. (examples: Netlify, Cloudcannon, etc)

Choice ParalysisWith the nature of the JAMstack not having a very defined list of applications to use, it could become difficult to choose which one to go with. My recommendation is to try a few different ones before you fully commit. If you really like the the plugin system and in WP, not a huge fan of getting into code and like the more ‚Äúbatteries included‚Äù option; I would recommend going with Gatsby If you feel you are more of a developer I would recommend NextJS. As for the Headless CMS I would recommend for you to either use Sanity.io if you would describe yourself as a developer and if not then I would recommend GraphCMS or Contentful. If you want to go super budget and with the least strings attached, I would ditch those Headless CMS providers and go with Markdown for your content and use Tina for your CMS.

This all might sound really interesting, but you or your client still would like to use WordPress as your CMS. Well you can have your cake and eat it too. Here are a few of the options you have:

WPGraphQL: With the GraphQL API for WordPress plugin you can decouple your site and join the modern web. The plugin allows you to connect to any of the popular JavaScript frameworks to power its content. So you can continue to use the CMS you love, but not have to battle with the frontend and leverage most or all of those frontend benefits I listed above. However, you will need to continue to maintain the security of your WP site along with its server environment. I have heard some developers locking their server down to specific IP addresses to try to reduce the risk.

WP2Static: A WordPress plugin that builds a static version of your site and can auto-deploy it when changes are made.

Strattic: They host the dynamic WordPress site for you (which they refer to as ‚Äústaging‚Äù) and you work with WordPress normally there. Then you choose to deploy, and they also host a static version of your site for you.

Shifter: Shifter hosts the WordPress site for you. You have two options: 1) run it headlessly (so you‚Äôre just hitting the API, REST or GraphQL, for data) or 2) run it statically (so when you have everything in WordPress where you want it, you deploy it, which creates a static version of your site, which they also host, or you can push it elsewhere e.g. Netlify).
